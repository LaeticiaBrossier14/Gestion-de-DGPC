# Extraction Automatique d'Informations à partir d'Appels d'Urgence Multilingues Low-Resource : État de l'Art 2025-2026
## Executive Summary
L'extraction automatique d'informations structurées (Lieu, Type d'incident, Urgence, Victimes) à partir d'appels d'urgence enregistrés en langues low-resource avec code-switching (Kabyle, Français, Darija) constitue un défi unique combinant défis technique, linguistique, et opérationnel. Ce rapport présente une analyse exhaustive de l'état de l'art 2025-2026, comparant sept approches architecturales majeures, identifiant les modèles et datasets optimisés pour cette tâche, et proposant une architecture de recherche à la fois innovante et réalisable pour une thèse de Master 2.
## 1. Approches Possibles : Classification Systématique
### 1.1 Pipeline Traditionnel : ASR → NLP → Classification
**Architecture** : Whisper/Cloud STT → Tokenization → BERT/SVM Classification

La première génération d'approches décompose le problème en étapes séquentielles. Un modèle ASR (Automatic Speech Recognition) convertit l'audio en texte, puis des méthodes NLP standard extraient l'information. Cette approche reste utilisée en production pour les appels d'urgence. [pmc.ncbi.nlm.nih](https://pmc.ncbi.nlm.nih.gov/articles/PMC12277377/)

**Implémentation concrète** : Le système ECA (Emergency Calls Assistant) de Attiah et al. 2025 suit ce pipeline : [pmc.ncbi.nlm.nih](https://pmc.ncbi.nlm.nih.gov/articles/PMC12277377/)
- Phase 1 : Google Cloud Speech-to-Text API (1.87s latence moyenne)
- Phase 2 : TF-IDF vectorization → SVM classification (0.05s)
- Résultat : 92.7% accuracy, 96.18% F1 score

**Points clés** :
- Les erreurs ASR se propagent directement à la phase NLP (goulot d'étranglement critique)
- Modularité : remplacer Whisper par des modèles spécialisés est trivial
- Déploiement : sans dépendance cloud (Whisper local possible)

**Limitations pour contexte low-resource** : [github](https://github.com/openai/whisper/discussions/1454)
- Degradation de la performance de language identification (LID) après fine-tuning sur Kabyle
- Besoin de catastrophic forgetting mitigation via LoRA

***

### 1.2 Fusion Multimodale : Audio + Texte avec Late-Fusion
**Architecture** : Deux pipelines parallèles (audio/texte) → Fusion logistique

Mirzaei et al. 2025 démontre que combiner signaux acoustiques ET linguistiques surpasse les approches unimodales, particulièrement pour détecter contradictions (ton calme + langage urgent). [thinkmind](https://www.thinkmind.org/articles/prediction_solutions_2025_1_10_30006.pdf)

**Composants** :

| Pipeline | Outil | Input | Sortie |
|----------|-------|-------|--------|
| Audio | openSMILE (ComParE config) | Raw audio | 6373 features (prosody, MFCC, HNR) |
| Audio | Random Forest | Features | Probability vector (urgent/non-urgent) |
| Texte | Whisper-medium | Audio | Transcription |
| Texte | MiniLM-L6-v2 | Text | 384-dim embeddings |
| Texte | 3-layer NN + softmax | Embeddings | Probability vector |
| Fusion | Logistic Regression | [Audio prob, Text prob] | Final urgency score |

**Performance** : 94% overall accuracy, **1.5% false-negative rate** (critique pour sécurité)

**Avantages pour Kabyle** : [thinkmind](https://www.thinkmind.org/articles/prediction_solutions_2025_1_10_30006.pdf)
- Compense bruit acoustique : même si Whisper a erreurs, prosody (pitch, speech rate) capture urgence
- Open-source tools : Ollama local + openSMILE → sovereignty des données
- Cost-sensitive learning : pénalise false negatives d'urgence

***

### 1.3 End-to-End Event Extraction : Audio → Événement Structuré
**Architecture** : Whisper-medium + Text2Event2 Transformer (finetuned avec LoRA)

Chaudhuri et al. 2025 propose approche directe : audio → représentation partagée → triggers + arguments événementiels [aclanthology](https://aclanthology.org/2025.xllm-1.24.pdf)

**Pipeline** :
1. Audio preprocessing : 16kHz mono, normalisation loudness (LUFS)
2. Whisper-medium : transcription bruitée (= données réalistes)
3. Text2Event2 (T5-based seq2seq) : trigger detection + role classification
4. LoRA fine-tuning : seuls ~4% des paramètres adaptés (critère : rang r=4, alpha=16)

**Défi spécifique** : Bruit ASR crée données d'entraînement corrompues
- Clean text (ACE05EN) : 72% F1 (événements complexes)
- Noisy ASR output : **44.6% F1** (rang 1er au XLLM Shared Task II)
- Gap : 27.4 points attributable exclusivement au bruit ASR

**Applicable au Kabyle** ?
- Avantage : évite pipeline, représentation unifiée
- Limitation : quasi aucune data annotée pour Kabyle events (faudrait créer dataset)
- Alternative : transfer learning depuis ACE05EN (anglais) vers Kabyle

***

### 1.4 Joint Slot-Intent Detection : Structure NLU Classique
**Architecture** : BERT/Contextual embeddings → BiLSTM-CRF → Intent + Slots

Muhammad et al. 2024 propose modèle multi-canal (MCNN) pour capturer contextual + non-contextual + syntactic features [pmc.ncbi.nlm.nih](https://pmc.ncbi.nlm.nih.gov/articles/PMC11623216/)

**Cas d'usage** : Dialogues assistants (ex : "Je dois rapporter un **accident à Béjaïa** avec **2 blessés**")
- Intent : reportEmergency
- Slots : location={Béjaïa}, injury_count={2}

**Performance sur ATIS** : 97.90% intent accuracy, 98.86% slot F1
**Performance sur SNIPS** : 98.88% intent, 97.07% slot F1

**Avantages pour appels d'urgence** :
- Format structuré natif (intent = emergency type, slots = {location, victim_count, urgency})
- Gestion d'erreurs bien comprise (BIO tagging, CRF decoder)

**Limitations** :
- Requiert transcription correcte (sensible aux erreurs ASR)
- Architecture complexe, nécessite données annotées pour fine-tuning
- Pas de support natif code-switching : deux approches nécessaires pour Kabyle+Français

***

### 1.5 Phoneme-based Direct Slot Filling : Audio → Phonème → Slots (Novel)
**Architecture** : Phoneme posteriors (HuBERT/Wav2Vec2) → Slot decoder (pas de transcription)

Wang et al. 2022 et Jung et al. 2024 explorent path radicalement différent : **éviter transcription entièrement** [aclanthology](https://aclanthology.org/2024.mrl-1.16.pdf)

**Concept** :
1. HuBERT large pré-entraîné extrait phoneme posteriors (représentation discrète)
2. Decoder direct : phoneme sequence → slot labels
3. Avantage : pas d'erreurs ASR transcription

**IPA Phonemic Representations**  : [aclanthology](https://aclanthology.org/2024.mrl-1.16.pdf)
- XPhoneBERT : 100+ langues avec input IPA (International Phonetic Alphabet)
- Performance cross-lingual réduit gap pour low-resource
- Kabyle + French + Arabic : alignement phonétique possible

**Applicabilité** :
- ✅ Évite erreurs ASR word-level
- ✅ Universal phoneme inventory (Kabyle/Arabic/French partagent patterns)
- ❌ Pas encore de dataset phoneme-to-slot pour urgences
- ❌ Nécessite phonemizer robuste pour code-switching

***

### 1.6 LLM Prompt-based Extraction : Few-Shot Prompting + Constrained Decoding
**Architecture** : Whisper → Text → LLM (in-context examples) → JSON (Outlines)

Suneet et al. 2024 et Polak et al. 2024 montrent LLMs capture sémantique complexe via prompts [aclanthology](https://aclanthology.org/2024.findings-acl.379.pdf)

**Pipeline** :
```
Prompt template:
"Given emergency call transcript, extract JSON with schema:
{location: string, incident_type: enum[fire|medical|accident|crime], 
 severity: enum[critical|high|medium], victims: {count: int, details: string}}"

In-context examples (few-shot):
- Example 1: "Il y a un incendie à l'hôpital de Béjaïa" → 
  {location: "Béjaïa", incident_type: "fire", severity: "critical", ...}
- Example 2: [2-3 autres exemples]

Test utterance:
"Help! Accident sur la RN5, 3 personnes blessées" →
JSON output (constrained par schema)
```

**Constrained Decoding**  : Outlines library [arxiv](https://arxiv.org/html/2501.10868v1)
- Compile JSON schema → finite state machine
- Génération parallèle avec masquage de tokens invalides
- Garantit output JSON valide (zéro hallucination de structure)

**Performance**  : [nature](https://www.nature.com/articles/s41467-024-45914-8)
- ChatGPT zero-shot : 85-92% extraction accuracy (dépend qualité prompt)
- Avec follow-up validation questions : 95%+
- APIE (Active Prompting IE) : sélection exemplaires incertains → +3-5% F1 [arxiv](https://arxiv.org/html/2508.10036v1)

**Avantages** :
- ✅ Zero-shot capable pour domaines non-vus
- ✅ Gère ambiguïtés, inférence implicite
- ✅ Erreurs Whisper moins impactantes (LLM corrige)

**Limitations** :
- ❌ Dépendance API (ChatGPT/Gemini) → privacy
- ❌ Coût par appel (utile si 10-100 appels/jour, non si millions/jour)
- ❌ Latence (10-30s vs. 2s pour pipeline)

**Pour déploiement local** : Ollama + Mistral/Llama 2
- Mistral 7B + Outlines : ~500ms/extraction
- Privacy preservée mais performance inférieure (-10-15% accuracy)

### 1.7 Neural-Symbolic Knowledge Graphs : Extraction → RDF → SPARQL Queries
**Architecture** : Whisper → ChatGPT triples → REBEL relation mapper → RDF/Wikidata → SPARQL

Santini et al. 2024 et Liu et al. 2024 intègrent structured knowledge [frontiersin](https://www.frontiersin.org/journals/computer-science/articles/10.3389/fcomp.2024.1472512/full)

**Pipeline** :
1. LLM extrait triples naturels : ("Béjaïa", location_of_incident, "Hôpital X")
2. REBEL (seq2seq) mappe relations → Wikidata properties
3. SBERT semantic filtering : cohérence triple
4. Sérialisation RDF + logical reasoning (owl2:sameAs, transitivity)

**Cas d'usage** : Requêtes complexes
- Q : "Tous incidents à Béjaïa impliquant feu depuis 24h ?"
- SPARQL : SELECT ?incident WHERE {?incident rdf:location ?loc . ?loc dc:title "Béjaïa" . ?incident dc:type wn:Fire}

**Avantages** :
- ✅ Interprétabilité maximale (graphe RDF queryable)
- ✅ Reasoning : inférer informations manquantes
- ✅ Longevity : données queryable des années après appel

**Limitations** :
- ❌ Latence multi-étape, complexité
- ❌ Ontologie design coûteux (2-3 mois pour domaine d'urgence)
- ❌ Moins adapté production temps-réel

## 3. Modèles et Outils Optimisés pour Low-Resource Multilingue
### 3.1 ASR pour Kabyle, Français, Darija
**Whisper Multilingual** [arxiv](https://arxiv.org/html/2503.23542v1)
- ✅ 680,000 heures pre-training (90+ langues)
- ✅ Robustesse bruit, accents
- ❌ Performance moindre low-resource après fine-tuning (LID dégradation)

**Fine-tuning Whisper sur Kabyle**  : [arxiv](https://arxiv.org/abs/2412.15726)
- Dataset : TutlaytAI/tamazight_asr  (Kabyle + Moroccan Amazigh) [huggingface](https://huggingface.co/datasets/TutlaytAI/tamazight_asr)
- Données : ~X heures (vérifier size exact)
- Méthode : Data generation (Swiss German approach) pour augmentation
- BLEU improvement : +15-20% vs. baseline Whisper

**Alternatives low-resource**: [github](https://github.com/openai/whisper/discussions/1454)
1. **Wav2Vec2 XLSR** : 53 languages, 56,000 heures
   - Fine-tuning + n-gram LM : -20% WER vs. baselines
   - Code-switching support meilleur que Whisper
   - Modèle : openreview.net ou HuggingFace

2. **Dialectal Arabic ASR** (Djanibekov 2024) [arxiv](https://arxiv.org/abs/2411.05872)
   - 17 pays covered, 11 variants
   - Code-switching ar-en/fr support
   - Models disponibles Hugging Face (Facebookresearch)

3. **DVoice Kabyle**  (wav2vec2.0 + CTC) [dataloop](https://dataloop.ai/library/model/aioxlabs_dvoice-kabyle/)
   - Pré-entraîné CommonVoice Kabyle
   - 16kHz mono input
   - Limité mais "Kabyle-native"

**Recommandation Master 2** :
- Primary : Fine-tune Whisper-medium avec TutlaytAI dataset
- Secondary : Wav2Vec2 XLSR para comparaison code-switching
- Augmentation : Créer ~500-1000 samples annotés appels d'urgence (crowdsourcing Algeria)

### 3.2 Modèles NLP pour Structure Extraction
**BERT-based Intent/Slot** [pmc.ncbi.nlm.nih](https://pmc.ncbi.nlm.nih.gov/articles/PMC11623216/)
- mBERT (multilingual BERT) : bon baseline
- CAMeLBERT (Arabic-specific) : meilleur pour arabe dialectal
- DziriBERT : Algérien optimisé (mais peu d'amélioration vs mBERT)

**Slot-Intent Architecture (SOTA)** :
```python
# MCNN-BiLSTM from  [arxiv](https://arxiv.org/abs/2407.21139)
class JointModel(nn.Module):
    def __init__(self):
        # 3 embedding channels
        self.bert_emb = BertEmbeddings()  # contextual
        self.w2v_emb = Word2Vec()  # semantic
        self.pos_emb = POSEmbeddings()  # syntactic
        
        # 3 CNN channels → max pooling
        self.cnn = MultiChannelCNN()
        
        # Shared BiLSTM
        self.bilstm = BiLSTM(hidden_size=256)
        
        # Decoder heads
        self.intent_head = Dense(num_intents)
        self.slot_head = CRF(num_slots)  # CRF pour séquence
```

**Performance** :
- Intent : 97-99% accuracy (aviation + multi-domain)
- Slots : 97-98.86% F1

### 3.4 Constrained Decoding pour JSON Output
**Outlines** [arxiv](https://arxiv.org/html/2501.10868v1)
- Compile JSON schema → FSM (finite state machine)
- Parallel execution avec masquage tokens
- Modèles supportés : Llama, Mistral, GPT (via API)

**Alternatives** :
- LlamaCpp : JSON schema support
- Guidance : Domain-specific language pour constraints
- XGrammar : EBNF grammar-based

**Exemple** :
```python
from outlines.models import transformers
import outlines.samplers as samplers

model = transformers.transformers("mistral-7b")
schema = {
    "location": "string",
    "incident_type": ["fire", "medical", "accident", "crime"],
    "severity": ["critical", "high", "medium"],
    "victim_count": "integer"
}
generator = outlines.samplers.json(model, schema)
result = generator("Accident RN5 Béjaïa, 3 blessés")
# Guaranteed valid JSON
```

### 5.2 Composants Détaillés
**Phase 1 : Audio Pre-processing**
```python
# Normalisation multi-language
import librosa, soundfile as sf

def preprocess_emergency_call(audio_path):
    y, sr = librosa.load(audio_path, sr=16000)
    
    # Loudness normalization (LUFS -16)
    S = librosa.feature.melspectrogram(y=y, sr=sr)
    loudness = librosa.power_to_db(S, ref=np.max)
    target_loudness = -16  # dB
    gain = target_loudness - loudness.mean()
    y_norm = np.power(10, gain/20) * y
    
    # Mono (déjà 16kHz)
    sf.write("preprocessed.wav", y_norm, sr)
    return y_norm, sr
```

**Phase 2A : Parallel ASR (Whisper fine-tuned)**
```python
from transformers import WhisperProcessor, WhisperForConditionalGeneration
import torch

# Whisper fine-tuned sur Kabyle
processor = WhisperProcessor.from_pretrained("TutlaytAI/whisper-medium-kabyle-ft")
model = WhisperForConditionalGeneration.from_pretrained("TutlaytAI/whisper-medium-kabyle-ft")

input_features = processor(y_norm, sampling_rate=16000, return_tensors="pt").input_features
predicted_ids = model.generate(input_features, task="transcribe", language="ky")
transcription = processor.batch_decode(predicted_ids, skip_special_tokens=True)[0]
# Output: "Ayaw aɛcident g'urar n Béjaïa, sǧent n yideɛ"
```

**Phase 2B : Parallel Audio Feature Extraction (Prosody)**
```python
import opensmile

# Acoustic features via openSMILE
extractor = opensmile.Smile(feature_set=opensmile.FeatureSet.ComParE_2016)
features = extractor.process_file(audio_path)
# Shape: (n_frames, 6373)

prosody = features[['F0semitoneFrom27.5Hz_sma', 
                     'jitterLocal_sma', 
                     'shimmerLocal_sma',
                     'MFCCs']].mean(axis=0)
# Capture stress indicators
```

**Phase 2C : Phoneme Extraction (HuBERT)**
```python
# HuBERT for phoneme posteriors (avoiding ASR word errors)
from transformers import Wav2Vec2Processor, HubertForCTC
import torch.nn.functional as F

processor = Wav2Vec2Processor.from_pretrained("facebook/hubert-large-ls960-pt")
model = HubertForCTC.from_pretrained("facebook/hubert-large-ls960-pt")

input_values = processor(y_norm, sampling_rate=16000, return_tensors="pt").input_values
logits = model(input_values).logits
phoneme_probs = F.softmax(logits, dim=-1)  # (1, T, 500 phonemes)
# Each timestep has probability distribution over phoneme inventory
```

**Phase 3 : Code-Switching Detection**

# Multi-label classification per word (from  [aclanthology](https://aclanthology.org/W15-0127.pdf))

**Phase 4 : Direct Phoneme→Slot (Novel)**
```python
# Bypass word-level ASR errors
# Map phoneme sequence → slot labels directly

class PhonemeSlotDecoder(nn.Module):
    def __init__(self, num_slots=10):
        super().__init__()
        self.hubert_encoder = HubertForCTC.from_pretrained(
            "facebook/hubert-large-ls960-pt"
        )
        # Freeze HuBERT, train only decoder
        for param in self.hubert_encoder.parameters():
            param.requires_grad = False
        
        self.slot_decoder = nn.Sequential(
            nn.Linear(1024, 512),
            nn.ReLU(),
            nn.Dropout(0.1),
            nn.Linear(512, num_slots)
        )
        self.crf = CRF(num_slots)
    
    def forward(self, audio, slot_labels=None):
        # HuBERT → hidden states (not CTC)
        hidden = self.hubert_encoder.extract_features(audio)
        # hidden shape: (batch, time, 1024)
        
        slot_logits = self.slot_decoder(hidden)
        # shape: (batch, time, num_slots)
        
        if slot_labels is not None:
            # Training with CRF
            loss = -self.crf(slot_logits, slot_labels)
            return loss
        else:
            # Inference
            predictions = self.crf.decode(slot_logits)
            return predictions

# Slots: {location, incident_type, severity, victim_count, urgency_level}
# Direct alignment: phoneme timestep → slot label
```

**Phase 5 : Multimodal Fusion for Urgency**
```python
# Combine phoneme slots + acoustic features
# Example: input "stress voice, calm words" → URGENT (acoustic wins)

class MultimodalFusion(nn.Module):
    def __init__(self):
        super().__init__()
        # Phoneme pathway
        self.phoneme_classifier = PhonemeSlotDecoder(num_slots=5)
        
        # Audio pathway
        self.prosody_classifier = nn.Sequential(
            nn.Linear(6373, 512),
            nn.ReLU(),
            nn.Linear(512, 128),
            nn.ReLU(),
            nn.Linear(128, 5)  # severity classes
        )
        
        # Late fusion
        self.fusion = nn.Sequential(
            nn.Linear(256, 128),
            nn.ReLU(),
            nn.Linear(128, 5)  # final classes
        )
    
    def forward(self, audio, phoneme_labels=None):
        # Phoneme branch
        phoneme_features = self.phoneme_classifier.encode(audio)
        
        # Prosody branch
        features = extract_opensmile(audio)
        prosody_features = self.prosody_classifier(features)
        
        # Concatenate
        combined = torch.cat([phoneme_features, prosody_features], dim=1)
        
        # Fusion
        output = self.fusion(combined)
        return output

fusion_model = MultimodalFusion()
urgency_scores = fusion_model(audio)
# Output: [location_distribution, incident_distribution, severity_distribution, ...]
```

## 8. Limitations et Directions Futures
### 8.1 Limitations Actuelles (2025)
1. **Kabyle resources scarce**
   - CommonVoice Kabyle : ~50 heures (petit)
   - Pas de emergency-call corpus public
   - Phoneme inventory Kabyle pas standardisé

2. **Code-switching understudied**
   - Majority papers : English ↔ high-resource
   - Kabyle ↔ French ↔ Darija : quasi-inexistant literature
   - Phoneme alignment may fail at code-switch boundaries

3. **Latency vs. Accuracy trade-off**
   - Phoneme approach : potentiellement slow (HuBERT large)
   - Need to optimize for <2s per-call (emergency critical)

4. **Annotation cost**
   - Creating 1000 Kabyle emergency calls : 3-6 mois (expensive)
   - Phoneme annotation (MFA) : error-prone if not careful

***

### 8.2 Directions Futures
**Short-term** (après Master 2) :
- Release dataset, benchmark future work
- Multilingual extension (Tamazight Moroccan, Amazigh)
- Speaker diarization (identify dispatcher vs. caller)
- Emotion recognition augmentation

**Medium-term** (2-3 ans après) :
- Cross-lingual transfer (French phoneme decoder → Kabyle)
- Real emergency center deployment + online learning
- Multimodal: Video (accident scene) + Audio fusion
- Legal/ethical impact assessment (bias in urgency prioritization)

**Long-term**:
- Unified Berber/Arabic/French foundation model (>1B parameters)
- Neurosymbolic reasoning (knowledge graphs + phoneme understanding)
- Real-time dialogue between dispatcher + AI assistant

L'extraction automatique d'informations à partir d'appels d'urgence en langues low-resource multilingues présente un **paysage riche d'approches** (2025-2026), allant du pipeline traditionnel aux architectures end-to-end sophistiquées. Pour une thèse Master 2 en Data Science, une **approche phoneme-aware hybrid** offre le meilleur équilibre :

- **Innovant** : phoneme-direct slot filling unexplored pour Kabyle
- **Réalisable** : leverage HuBERT pre-trained, ~500-1000 annotations manuelles
- **Impactant** : directly applicable emergency response Algérien
- **Privé & Local** : 100% deployable sans cloud

Les bases (Whisper, HuBERT, Wav2Vec2, constrained decoding) existent; l'effort recherche réside dans **optimisation contextuelle** et **dataset creation** pour domaine critique underserved. Avec 12 mois focalisés et dataset public release, contribution sera à la fois **scientifiquement novel** et **socialement utile**.

Question **fondamentale**. C'est la différence entre un modèle qui **triche** et un modèle qui **comprend**. Laisse-moi t'expliquer.

## Le risque : "Right answer, wrong reason"

Imagine ton dataset a 500 exemples. Si 80% des fois que "times" apparaît c'est `fire_building`, le modèle apprend un **raccourci** :

```
"times" → fire_building (80% du temps dans le dataset)
```

Mais "times" veut dire "feu" en général. Ça pourrait être :
- `fire_building` (incendie maison)
- `fire_forest` (feu de forêt)
- `fire_vehicle` (voiture en feu)

Et "tamdint" (ville) n'a rien à voir avec le feu. Le modèle ne les confondra pas **si le dataset est bien construit**.

## Comment le modèle apprend RÉELLEMENT

Un LLM ne fait pas `mot → label`. Il fait `contexte entier → label`. Regarde :

```
Exemple 1: "times tech3el g wexxam g Tichy"
           feu   brûle   dans maison  à Tichy
           → fire_building ✅ (feu + maison)

Exemple 2: "times tech3el g tagant n Yakouren"  
           feu   brûle   dans forêt  de Yakouren
           → fire_forest ✅ (feu + forêt)

Exemple 3: "times tech3el g tomobil g RN9"
           feu   brûle   dans voiture sur RN9
           → fire_vehicle ✅ (feu + voiture)

Exemple 4: "yella l'accident g tamdint n Béjaïa"
           il y a accident dans ville  de Béjaïa
           → accident_vehicular ✅ (accident + ville, pas de feu)
```

Le modèle apprend que :
- `times` + `wexxam` → `fire_building`
- `times` + `tagant` → `fire_forest`
- `times` + `tomobil` → `fire_vehicle`
- `tamdint` = contexte lieu, pas un incident

C'est le **contexte combinatoire** qui donne le sens, pas le mot seul. Et c'est exactement ça la force d'un LLM — il comprend le contexte.

## Mais le vrai danger : dataset biaisé

Le problème n'est **pas** que le modèle confond "times" et "tamdint". Le problème c'est quand ton dataset a :

```
DATASET BIAISÉ (MAUVAIS):
├─ 200 exemples "fire_building"   (40%)
├─ 150 exemples "accident_vehicular" (30%)
├─ 50 exemples "medical_emergency" (10%)
├─ 10 exemples "drowning"  (2%)
├─ 5 exemples "hazmat"  (1%)
└─ 85 exemples autres (17%)

Résultat: le modèle répond "fire_building" pour tout ce qu'il comprend pas
         parce que c'est la réponse la plus fréquente → RIGHT BY CHANCE
```

## Les 3 protections contre le "right by chance"

### Protection 1 : Diversité des exemples par type

Pour chaque `incident_type`, tu as besoin de **minimum 30-50 exemples** avec des formulations variées :

```
fire_building (50 exemples):
├─ "times tech3el g wexxam"          (kabyle pur)
├─ "l7riq g lbatima"                 (darija)
├─ "yella incendie g l'appartement"  (code-switch FR)
├─ "lḥriq yella g lmadrasa"         (kabyle + arabe)
└─ "ça brûle dans la maison"        (français pur)

fire_forest (50 exemples):
├─ "times g tagant"
├─ "l7riq g djebel"
├─ "feu de forêt sur la montagne"
└─ "tech3el tagant g Yakourene"
```

Le modèle voit que **le même mot** (`times`) mène à des types différents selon le **contexte** → il ne peut pas tricher avec un raccourci.

### Protection 2 : Contre-exemples explicites

Ajouter des exemples où des mots similaires mènent à des résultats **différents** :

```json
{
  "input": "times tech3el g tomobil",
  "output": {"incident_type": "fire_vehicle"}       ← feu + véhicule
}
{
  "input": "teqleb tomobil g tamdint",
  "output": {"incident_type": "accident_vehicular"} ← renversé + véhicule (pas de feu!)
}
{
  "input": "yella times g tamdint bessah machi g wexxam, g tagant qrib",
  "output": {"incident_type": "fire_forest"}         ← feu PRÈS de la ville mais DANS la forêt
}
```

Ce dernier exemple est crucial — le modèle apprend que `tamdint` ici est contexte géographique, le feu est dans `tagant`.

### Protection 3 : Le champ `notes` comme "chain-of-thought" léger

Tu peux enrichir ton format d'entraînement avec un champ qui **explique le raisonnement** :

```json
{
  "input": "times tech3el g wexxam g Tichy sin n yirgazen jerhen",
  "output": {
    "incident_type": "fire_building",
    "location": "Tichy",
    "victims_count": 2,
    "injuries_severity": "minor",
    "fire_present": "yes",
    "notes": "times=feu, wexxam=maison → fire_building. jerhen=blessés légers → minor"
  }
}
```

Ce `notes` force le modèle à **verbaliser son raisonnement**. En inférence, si le modèle écrit `"notes": "times=feu, tomobil=véhicule → fire_building"` alors que `tomobil` aurait dû donner `fire_vehicle`, tu vois immédiatement l'erreur de raisonnement.

C'est une forme light de **Chain-of-Thought** (CoT), prouvé par la recherche pour améliorer la précision des LLM.

## Comment vérifier que le modèle ne triche pas

Après entraînement, tu testes avec des **paires adversariales** :

```
TEST 1 (minimal pair — un seul mot change):
├─ "times g wexxam"   → attend: fire_building
├─ "times g tagant"   → attend: fire_forest
├─ "times g tomobil"  → attend: fire_vehicle
└─ Si les 3 sont corrects → le modèle utilise le contexte, pas juste "times"

TEST 2 (même contexte, incident différent):
├─ "yeqleb tomobil g Tichy"    → attend: accident_vehicular (pas de feu)
├─ "tech3el tomobil g Tichy"   → attend: fire_vehicle (feu)
└─ Si correct → le modèle distingue "yeqleb" (renverser) de "tech3el" (brûler)

TEST 3 (ambiguïté volontaire):
├─ "yella kra g tamdint"   → attend: "unknown" + notes indiquant ambiguïté
└─ Si le modèle dit "unknown" plutôt que deviner → il est honnête
```

## Résumé en une image

```
MOT SEUL          CONTEXTE                    RÉSULTAT

"times"     +      "wexxam" (maison)      →   fire_building
```

Et les 3 protections (diversité, contre-exemples, notes CoT) garantissent qu'il raisonne correctement plutôt que de deviner. C'est exactement ça qu'on construira quand on migrera les 5000 entrées vers le nouveau format — on s'assurera que chaque type a assez de variantes et de contre-exemples.Exactement. ## La donnée synthétique n'est pas là pour "avoir plus de données"

C'est là pour **corriger les déséquilibres** que les données réelles ont forcément :

```
DONNÉES RÉELLES DGPC (ce que tu as):
├─ accident_vehicular    ███████████████████  180  (36%)  ← Béjaïa = beaucoup de routes
├─ fire_building         ██████████           95   (19%)
├─ medical_emergency     ████████             70   (14%)
├─ fire_forest           ██████               55   (11%)  ← saisonnier (été)
├─ assault_violence      ███                  25   (5%)
├─ drowning              ██                   15   (3%)   ← saisonnier (plage)
├─ structural_collapse   █                    8    (1.6%) ← rare
├─ hazmat                █                    5    (1%)   ← très rare
├─ lost_person           █                    5    (1%)
└─ fire_vehicle          ░                    2    (0.4%) ← quasi absent
                                             ───
                                             500 total

PROBLÈME: Le modèle verra "accident_vehicular" 90× plus que "fire_vehicle"
          → Il apprend très bien les accidents, devine au hasard pour le reste
```

## L'augmentation synthétique cible PRÉCISÉMENT les trous

```
APRÈS AUGMENTATION CIBLÉE:
├─ accident_vehicular    180 réels + 0 synth  = 180  ← déjà assez
├─ fire_building         95  réels + 0 synth  = 95   ← OK
├─ medical_emergency     70  réels + 10 synth = 80   ← léger boost
├─ fire_forest           55  réels + 25 synth = 80   ← augmenté
├─ assault_violence      25  réels + 45 synth = 70   ← bien augmenté
├─ drowning              15  réels + 55 synth = 70   ← bien augmenté
├─ structural_collapse   8   réels + 62 synth = 70   ← fortement augmenté
├─ hazmat                5   réels + 65 synth = 70   ← fortement augmenté
├─ lost_person           5   réels + 65 synth = 70   ← fortement augmenté
└─ fire_vehicle          2   réels + 68 synth = 70   ← fortement augmenté
                                               ───
                                               ~855 total, équilibré
```

Chaque type a **minimum 70 exemples**, avec des formulations variées (kabyle pur, code-switch, français, darija).

## Les 2 règles d'or

**Règle 1 : Ne jamais augmenter ce qui est déjà assez représenté**

Si tu as 180 exemples d'accidents, générer 200 de plus = bruit. Le modèle les connaît déjà.

**Règle 2 : Chaque exemple synthétique doit apporter de la VARIÉTÉ, pas de la quantité**

```
MAUVAIS (copier-coller):
├─ "yella hazmat g Béjaïa"
├─ "yella hazmat g Akbou"
├─ "yella hazmat g Tichy"        ← même structure, juste le lieu change
└─ Le modèle apprend 1 pattern, pas 65

BON (variété):
├─ "yella gaz yettsarrab g l'usine n Akbou"        (fuite gaz, usine)
├─ "l'camion n lmazout yeqleb g RN12"               (citerne renversée)
├─ "iruḥ le produit chimique g lbḥer n Tichy"      (déversement mer)
├─ "ça sent le gaz dans tout l'immeuble bessah"     (gaz résidentiel)
├─ "yella chlore g la piscine, les enfants malades"  (chlore piscine)
└─ Le modèle apprend 5 scénarios DIFFÉRENTS de hazmat
```

## Résumé

| Ce que beaucoup pensent | La réalité |
|---|---|
| "Plus de données = mieux" | Plus de données **équilibrées et variées** = mieux |
| Data augmentation = ×10 le dataset | Data augmentation = combler les trous |
| Synthétique partout | Synthétique **uniquement** là où le réel manque |

C'est exactement cette logique que tu appliqueras quand on migrera les 5000 entrées : d'abord compter la distribution par type, identifier les trous, puis générer avec Gemini uniquement pour les types sous-représentés, avec un maximum de variété dans les formulations.+

## Phase 3 : Le protocole de génération (LLM + Humain)

Tu as raison, il faut les deux. Voici le workflow exact :

```
ÉTAPE 1: L'HUMAIN définit le QUOI
════════════════════════════════════
Toi tu décides:
- "Je veux 50 exemples de drowning"
- "Variés: plage, rivière, piscine, oued, barrage"
- "Lieux: Tichy, Aokas, Souk El Tenine, Ighram"
- "Langues: 60% kabyle, 30% code-switch, 10% français"
- "Victimes: enfants, adultes, groupes"

ÉTAPE 2: LE LLM génère le COMMENT
════════════════════════════════════
Tu donnes à Gemini un prompt structuré:

  "Génère 10 phrases d'appel d'urgence pour une NOYADE 
   à la PLAGE DE TICHY en dialecte kabyle de Béjaïa.
   
   Règles:
   - Utilise le code-switching naturel bejaoui
   - Varie: nombre de victimes, âge, gravité
   - Varie: le style (paniqué, calme, témoin, famille)
   - Inclus les connecteurs: bessah, wesh, za3ma, wallah
   - Chaque phrase UNIQUE, pas de copier-coller
   
   Format par phrase:
   {
     "input": "la phrase kabyle",
     "incident_type": "drowning",
     "location": "Tichy",
     "victims_count": X,
     "injuries_severity": "...",
     "notes": "explication des mots clés kabyles utilisés"
   }"

Gemini génère:
1. "Wesh arwah-d s zerb! Yeghraq yiwen n waqcic g lbḥer n Tichy!"
2. "3edjled l'ambulance! Sin n tullas ghraqent g la plage Tichy bessah!"
3. "Yella urgaz yeghraq g lebḥer, ur d-yeffiɣ ara, dagi g Tichy!"
...

ÉTAPE 3: L'HUMAIN valide le RÉSULTAT
════════════════════════════════════════
Pour chaque phrase générée, tu vérifies:

  ✅ "Wesh arwah-d s zerb! Yeghraq yiwen n waqcic g lbḥer n Tichy!"
     → Naturel, un vrai bejaoui dirait ça ✓
     → "yeghraq" = correct pour noyade ✓
     → Grammaire kabyle correcte ✓
     → APPROUVÉ
  
  ❌ "Il y a une personne qui se noie dans la mer de Tichy SVP"
     → Trop français, aucun bejaoui parle comme ça au téléphone
     → REJETÉ ou REFORMULÉ
  
  ⚠️ "Yeghraq urgaz g la piscine n Tichy"
     → OK linguistiquement mais... Tichy n'a pas de piscine publique connue
     → MODIFIÉ: changer lieu ou enlever "piscine"
```
